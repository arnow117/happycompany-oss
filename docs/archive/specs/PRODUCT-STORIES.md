## 故事 1：管理员的日常 — 发布和安装新 App

**角色**：平台管理员（小王）
**场景**：医疗设备公司，示例医疗器械

小王刚用 Claude Code 开发了一个新的「器械报修」App。他把代码推到了公司仓库，现在要把这个 App 上线到内部 bot。

1. 打开管理后台 `http://localhost:3000`
2. 点击「发布新版本」，填入：
   - App 名称：`device-repair`
   - 版本：`v1.0`
   - 源码目录：`/home/wang/apps/device-repair/v1.0`
   - 描述：`医疗器械报修流程管理`
3. 点击发布。后台自动将源码复制到 registry，并校验一致性（README ↔ SKILL.md ↔ CLAUDE.md）
4. 在「已发布 App」列表看到 `device-repair v1.0`，状态显示 0 个一致性警告
5. 点击「安装到工作区」，选择工作目录
6. 安装完成。后台自动清除了该 bot 的所有活跃 session，确保新 skill 立即生效
7. 第二天同事在钉钉群里 @bot 提问「这台呼吸机需要报修」，bot 已经知道怎么处理了

## 故事 2：知识库的沉淀 — 从聊天记录到可检索知识

**角色**：领域专家（李工）
**场景**：把日常问答中的知识点系统化

李工是公司的资深工程师，每天在钉钉群里回答同事的技术问题。他发现很多问题反复出现。

1. 管理员在后台安装了「知识管理」App（kb-management）
2. 李工在群聊里输入 `/kb-ingest`，然后粘贴一段关于「呼吸机参数校准」的详细回复
3. bot 回复：「知识卡片已入库：呼吸机参数校准（#KB-0042）」
4. 一周后，新同事在群里 @bot 问「呼吸机参数怎么校准」
5. bot 自动检索知识库，回复：「根据知识库 #KB-0042，呼吸机参数校准流程如下...」

知识不再是散落在聊天记录里的碎片，而是可检索、可引用的结构化资产。

## 故事 3：Python 专家能力 — 复杂计算交给专业工具

**角色**：数据分析师（张姐）
**场景**：需要 bot 帮忙做统计计算

张姐是公司的数据分析师，需要 bot 能帮她做一些简单的统计计算。

1. 管理员发布了一个 Python App `stats-tool`，包含 `bin/run` 入口和 `src/calc.py` 计算脚本
2. 安装到工作区后，平台自动检测到这是 Python App（通过 `detectAppLanguage()`）
3. 张姐在群里 @bot：「帮我算一下这组数据的均值和标准差：[12.3, 15.1, 11.8, 14.5]」
4. bot 识别到需要统计计算，调用 `stats-tool` 的 CLI 入口
5. 返回结果：「均值 = 13.43，标准差 = 1.38」

Python 的数值计算能力和 TypeScript 的平台编排能力各取所长，通过 CLI 接口无缝协作。

## 故事 4：安全第一 — 管理员的安全职责

**角色**：安全管理员
**场景**：确保平台安全合规

安全管理员每月做一次安全审查。

1. 查看管理后台的 API 鉴权配置，确认 Bearer token 已启用
2. 检查 webhook 端点：飞书和钉钉的原始 payload 都经过 Zod schema 校验，恶意格式化数据会被拒绝
3. 检查文件访问：所有文件 API 都经过 `safePath()` 路径穿越防护
4. 检查 App 一致性：调用 `GET /api/admin/apps/:name/consistency` 逐个审计
5. 发现某个 App 的 SKILL.md 名称和 README 标题不匹配，标记为待修复
6. 检查 session 管理：确认安装 App 后旧 session 已清除，不会泄露过期信息

## 故事 5：迭代闭环 — 从使用数据到产品改进

**角色**：产品经理 + 管理员
**场景**：基于真实使用数据改进 bot

产品经理想了解 bot 的实际使用情况，决定下一步改进方向。

1. 打开管理后台「使用分析」页面
2. 查看过去 7 天的消息趋势：日均 120 条消息，高峰在工作日下午 3 点
3. 发现「器械报修」skill 的调用频率最高（占 35%），但错误率 12%
4. 发现「知识查询」skill 几乎没被使用
5. 让管理员通过 Skill 管理 API 更新「器械报修」的 SKILL.md，优化提示词
6. 两周后复查，错误率降到 4%

使用数据不是用来汇报的，而是用来指导下一次迭代的。

## 故事 6：多 Bot 协作 — 一个群里的专业分工

**角色**：一线员工
**场景**：不同问题找不同的 bot

公司有三个 bot：示例医疗助手（通用）、报修助手（器械维护）、知识助手（文档查询），都在同一个钉钉群里。

1. 员工 @报修助手：「三号呼吸机报警代码 E-042 怎么处理」
   - 只有报修助手回复，其他 bot 不打扰
2. 同一个人 @知识助手：「公司报修流程文档在哪里」
   - 知识助手回复，附上文档链接
3. 员工没 @任何人：「明天有谁值班」
   - 没有 bot 回复（群聊中必须 @才会响应）

每个 bot 的能力描述来自 workdir/CLAUDE.md，员工 @ 的时候能准确找到对的专家。
