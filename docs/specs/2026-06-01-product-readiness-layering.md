# Product Readiness Layering

Date: 2026-06-01

## Background

HappyCompany already has enough visible surface area that users can confuse three different states:

- production links that can operate real tenant data;
- builder and verification tools used to create or test digital employees;
- previews that visualize traces or blueprints but are not yet an active runtime.

This creates a product risk: the UI can imply "active workflow automation" when the current implementation is actually "trace observation plus process guidance".

## Goal

Make capability maturity explicit in the navigation, page titles, and primary copy before adding deeper workflow runtime behavior.

The first slice is a product contract cleanup, not a workflow-engine implementation.

## Capability Tiers

### Production

Production features can be used as part of the live employee loop:

- Chat
- Sessions
- people and employee management
- people binding
- entry routing
- knowledge base

### Build

Build features prepare or validate production behavior, but are not themselves the live channel:

- digital employee Builder
- Harness
- skills marketplace
- employee network

### Preview

Preview features expose design intent, traces, or experimental concepts:

- workflow blueprints
- memory inspection

Preview surfaces must avoid language that implies an active state machine, autonomous workflow scheduler, or human approval gate unless that backend exists.

### Ops

Ops features configure and observe the platform:

- Dashboard
- Stats
- Config
- Agent Status
- Scheduler

## Five-Layer Iteration Plan

### L1. Product Promise Layer

Status: implemented.

1. Re-group sidebar navigation around the four tiers.
2. Add compact tier badges in expanded navigation.
3. Collapse the previous workflow/blueprint/orchestration wording into one user-facing concept: `编排`.
4. Rename Harness page to "验收 Harness" and clarify reports are trace-based.
5. Rename Builder page to "数字员工 Builder" and clarify natural language fills a structured employee draft.
6. Mark form orchestration builder as a preview rather than an active publisher.

### L2. Capability Registry Layer

Status: implemented in this iteration.

Add a backend capability report and Web page so each digital employee exposes:

- prompt source: YAML `systemPrompt` and workspace `CLAUDE.md`;
- skills: installed state, display name, tool count, permission state;
- tools: registration state, risk level, permission state;
- handoff targets: existence and display name;
- workspace: relative path, absolute path, `CLAUDE.md` health;
- MCP boundary: platform MCP visible, business MCP hidden by default, business entry through `run_skill`.

### L3. Builder Assembly Layer

Status: implemented in this iteration.

Builder must present skill/tool/target selection as capability assembly rather than raw textarea editing. Publish remains blocked until validation and harness pass.

The Builder now exposes a draft capability preview generated by the backend, using the same capability report shape as published employees. The review panel shows the draft's skill count, tool count, handoff targets, workspace/CLAUDE.md state, and risk count before validation/test/publish.

### L4. Orchestration Runtime Layer

Status: updated on 2026-06-02.

Product decision: only `编排` remains as the user-facing concept. The old `工作流蓝图` and `工作流` product surfaces are removed from navigation and copy.

The Orchestration page shows only:

- observed routing traces;
- employee handoff/decomposition;
- recent collaboration execution records.

This iteration connects the existing Harness `StepRunner` state machine to Web:

- `POST /api/admin/harness/run-step` dispatches one step through `MessageIngressRuntime`.
- `GET /api/admin/harness/step-runs` lists persisted in-memory StepRuns.
- `/api/orchestration/traces` is the only Web-facing API for this surface.
- `/orchestration` is the single route and navigation entry. The old `/workflows` route is removed.

This is intentionally a trace-oriented orchestration surface, not yet a proactive scheduler or approval runtime.

### L5. Harness And Release Gate Layer

Status: implemented as a minimal release-gate bridge in this iteration.

Harness should be usable as the shared test entry for Web/DingTalk/runtime stories. Builder-generated smoke harness is allowed for MVP, but must be clearly labeled as simulated until it runs through the real runtime.

Harness now exposes both suite-level Trace reports and long-task StepRun state, creating a shared test entry for runtime stories and future release gates.

## Deferred Work

- Real orchestration persistence and state machine.
- Human gate / approval checkpoint runtime.
- Scheduled proactive orchestration execution.
- Builder-generated end-to-end harness cases that run through the same runtime as Web/DingTalk.
- Capability registry page that explains exact skills, tools, MCP servers, and permissions per employee.
